# 代码审查流程与清单

## 1. 审查流程

```
开发者提交 PR → 自动 CI 检查 → CTO/架构师审查 → 反馈修改 → 重新审查 → 合并
```

### 1.1 审查角色分工
- **CTO**: 负责核心模块、架构变更、公共组件的审查
- **架构师**: 负责技术选型、设计模式、性能优化相关审查
- **高级开发者**: 可参与常规功能模块审查（未来扩展）

### 1.2 审查时效要求
- **普通 PR**: 24 小时内完成审查
- **紧急修复**: 4 小时内完成审查
- **重大重构**: 48 小时内完成审查（需架构师参与）

## 2. 自动化检查清单 (CI 自动执行)

✅ **必须全部通过才能进入人工审查**

- [ ] 编译成功 (无错误/警告)
- [ ] 所有单元测试通过
- [ ] 测试覆盖率 ≥70% (核心模块 ≥85%)
- [ ] Ktlint 代码风格检查通过
- [ ] Detekt 静态分析无严重问题
- [ ] 无新增安全漏洞 (依赖检查)

## 3. 人工审查清单

### 3.1 代码质量
- [ ] 遵循 Kotlin 编码规范 (命名、格式、注释)
- [ ] 函数长度合理 (<50 行为宜)
- [ ] 类职责单一 (SRP)
- [ ] 避免重复代码 (DRY)
- [ ] 适当的错误处理 (try-catch/Result)
- [ ] 日志使用合理 (Timber/Logger)

### 3.2 架构规范
- [ ] 遵循 MVVM + Clean Architecture 分层
- [ ] ViewModel 不包含 Android 框架依赖
- [ ] UseCase 保持业务逻辑纯粹
- [ ] Repository 抽象正确 (接口 + 实现分离)
- [ ] 依赖注入使用 Hilt/Koin 正确
- [ ] 数据流清晰 (StateFlow/LiveData 使用恰当)

### 3.3 测试覆盖
- [ ] 新增代码有对应单元测试
- [ ] 边界条件有测试用例
- [ ] 异常场景有测试覆盖
- [ ] 测试命名清晰 (given_when_then)
- [ ] Mock 使用合理 (不过度 Mock)

### 3.4 性能与安全
- [ ] 无主线程 IO 操作
- [ ] 适当使用协程作用域 (避免内存泄漏)
- [ ] 敏感数据不硬编码 (使用 BuildConfig/加密)
- [ ] 网络请求有超时设置
- [ ] 图片加载有缓存策略
- [ ] 列表有分页/懒加载

### 3.5 文档与注释
- [ ] 公共 API 有 KDoc 注释
- [ ] 复杂逻辑有注释说明
- [ ] TODO/FIXME 标注清晰
- [ ] CHANGELOG 已更新 (如适用)

## 4. 审查意见规范

### 4.1 意见分类
- **[BLOCKER]**: 必须修复才能合并 (架构错误/安全漏洞)
- **[MAJOR]**: 强烈建议修复 (性能问题/代码异味)
- **[MINOR]**: 可选优化 (命名建议/格式微调)
- **[QUESTION]**: 需要讨论澄清

### 4.2 意见格式示例
```kotlin
// ❌ 不好
"这里有问题"

// ✅ 好
"[MAJOR] 建议在 ViewModel 中使用 StateFlow 而非 LiveData，
原因：StateFlow 支持背压且与协程集成更好。
参考：https://kotlinlang.org/docs/flow-state-flow.html"
```

## 6. 合并策略

- **分支保护**: main/develop 分支开启保护，禁止直接推送
- **审查人数**: 至少 1 人批准 (核心模块需 2 人)
- **合并方式**: 
  - feature → develop: Squash Merge
  - develop → main: Merge Commit (保留历史)
- **合并后**: 自动删除源分支

## 7. 常见拒绝原因

| 原因 | 说明 |
|------|------|
| 测试不足 | 覆盖率低于阈值或缺少关键场景测试 |
| 架构违规 | 违反分层原则或引入循环依赖 |
| 性能风险 | 存在明显性能隐患 (主线程 IO/内存泄漏) |
| 代码重复 | 已有相同/相似功能未复用 |
| 文档缺失 | 公共 API 无注释或变更未更新文档 |
