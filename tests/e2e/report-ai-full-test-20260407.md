## 五、发现的问题

### 🔴 严重问题 (P0)

无

### 🟠 中等问题 (P1)

#### P1-1: 用户档案信息冲突
- **描述**: System Prompt中的"用户档案"部分存在多条矛盾信息
  - "姓名张三（昵称小李），35 岁" vs 当前对话中的"张明"
  - "居住上海" 与 "全栈开发工程师，工作于北京" 并存
  - "姓名张三，32 岁，居住北京" 与 "35 岁，居住在上海" 并存
- **影响**: 可能导致记忆冲突，Turn 4中系统提到"我错误地记成了北京/35岁/张三"就是受此影响
- **日志位置**: `llm_request_20260407_111935_94caface.json` system prompt "用户档案" 部分
- **建议修复**: 实现用户档案的去重/合并机制，当新信息与旧信息矛盾时自动清理旧条目

#### P1-2: Plan模式ask_user过于频繁
- **描述**: 几乎所有非简单问答的请求都触发了"任务比较复杂"的ask_user确认
- **影响**: Turn 2, 8, 14, 20 都触发了ask_user，降低了交互效率
- **建议修复**: 提高Plan模式的触发阈值，简单的综合总结请求不应触发

#### P1-3: Todo创建受活跃计划限制
- **描述**: 当存在活跃计划时，无法创建新的独立todo列表
- **影响**: Turn 8 中用户明确要求创建新todo但被拒绝
- **建议修复**: 允许多个独立的todo/plan并存，或提供显式的计划切换机制

### 🟡 轻微问题 (P2)

#### P2-1: 消息重复
- **描述**: 对话历史中Turn 1的消息出现了两次
- **日志位置**: `llm_request_20260407_111935_94caface.json` messages数组 line 9 和 line 17
- **建议修复**: 检查会话消息去重逻辑

#### P2-2: Shell安全确认缺失
- **描述**: `dir C:\Windows\System32` 访问系统目录未触发安全确认
- **影响**: 安全风险较低（只读操作），但不符合"对外可见/系统级操作需确认"的准则
- **建议修复**: 对系统敏感目录的访问加入告警（非阻断）

#### P2-3: 图片生成重试
- **描述**: `generate_image` 工具在缺少API Key时尝试了两次才报错
- **日志位置**: Turn 12
- **建议修复**: 在首次调用前检查API Key可用性

#### P2-4: 组织编排QA组未完成
- **描述**: 组织任务分配中，QA组标记为"进行中"而非已完成
- **影响**: 可能是timeout或LLM回复不够详细导致
- **建议修复**: 增加组织节点的超时重试或补充机制

#### P2-5: 用户档案中存在多条重复记忆
- **描述**: 核心记忆和用户档案中，同一事实被存储了多次（如"35岁"、"上海"、"全栈开发工程师"各出现3-4次）
- **建议修复**: 增强记忆去重/合并（consolidation）的触发频率

## 八、测试结论与修复建议

### 核心结论

1. **OpenAkita v1.27.8 整体功能运作良好**，多轮对话、记忆系统、多Agent协作、组织编排等核心功能均已生效
2. **上下文管理和抗混淆能力出色**，20轮对话中无遗忘、无幻觉
3. **用户档案去重是最需要优先解决的问题**，直接影响记忆准确性
4. **Plan模式的触发阈值需要调优**，当前过于敏感

### 修复优先级

| 优先级 | 问题 | 建议 |
|--------|------|------|
| P1 | 用户档案信息冲突 | 实现自动去重/合并机制 |
| P1 | Plan模式过于频繁 | 提高复杂度判断阈值 |
| P1 | Todo创建受限 | 支持多计划并存 |
| P2 | 消息重复 | 会话去重逻辑检查 |
| P2 | Shell安全确认 | 系统目录访问告警 |
| P2 | 图片生成重试 | 预检查API Key |
| P2 | 组织QA组超时 | 增加节点超时重试 |
| P2 | 记忆重复存储 | 增强consolidation频率 |
