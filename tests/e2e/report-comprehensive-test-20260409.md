## 四、发现的问题 — 严格审查（对比 Claude Code 设计）

> 以下对每个问题逐一审查：**是否真需要修改？根因是否正确？提议的修复是否合理？**
>
> 参考设计：Claude Code 的上下文管理完全**不存在**话题切换检测机制。
> 它使用纯 token 阈值触发的 `compactConversation` — 把旧消息整体摘要为一条 summary，
> 附带 `createCompactBoundaryMessage`。没有试图"智能检测话题"，也没有"边界标记指令"。
> 整个对话历史要么完整保留，要么统一压缩为摘要。

---

### 问题 1：话题切换边界标记导致 LLM 忽略历史上下文

**是否必须修改？** ✅ **是，这是一个真实 Bug**

**根因审查 — 原报告的根因基本正确，但不够精确：**

原报告说"边界标记文案导致 LLM 忽略"。更精确的分析如下：

1. `agent.py:3248` 的话题检测守卫 `if session and len(session_messages) >= 4:` **没有通道限制**，
   但 `_detect_topic_change` 的 docstring 写着"仅在 IM 通道调用"。这是一个**守卫遗漏** —
   Desktop 聊天也会触发该逻辑，违反了设计意图。

2. 即使只在 IM 通道生效，核心设计也有缺陷：当用户说"回到之前的话题"时，系统把它当作 `NEW` 话题
   （因为它跟最近 6 条消息不相关），然后注入 `[上下文边界] ...请优先关注边界之后的内容`。
   这条指令直接导致 LLM 忽视边界之前的消息。

3. 边界注入后，`context_manager.py:508-560` 的 `_compress_across_boundary` 会在后续轮次
   以 25% 的比例（`context_boundary_compression_ratio=0.25`）激进压缩边界前的消息，进一步
   丢失事实。

**对比 Claude Code 的设计**：Claude Code 的 `compactConversation` 完全不做话题检测。它用纯 token
数触发 compact（`shouldAutoCompact` 基于 token threshold），把**所有旧消息**统一摘要。
compact 后的 boundary marker (`createCompactBoundaryMessage`) 只是一个纯结构标记，**不包含
任何指导 LLM 行为的文案**。这是正确的设计 — 上下文管理不应该在 messages 中注入影响 LLM 推理
的指令。

**结论**：问题 1 是一个**架构级设计缺陷**，不是微调文案就能修的。

**修复方案（分级）**：

| 方案 | 描述 | 评价 |
|------|------|------|
| ❌ 改文案 | 把"请优先关注边界之后"改成中性措辞 | **垃圾方案** — 治标不治本。只要 messages 中有人工注入的指令性内容就是 prompt injection |
| ❌ 关键词排除 | 检测"回忆/之前说过"等关键词时不注入边界 | **垃圾方案** — 枚举关键词是脆弱的，用户可能用任意措辞引用早期内容 |
| ⚠️ 加通道守卫 | 在 `agent.py:3248` 加 `channel == "im"` 限制，Desktop 不触发话题检测 | **应急可行** — 符合 docstring 的原始设计意图，但 IM 通道仍有此问题 |
| ✅ 移除边界指令 | 保留 `[上下文边界]` 结构标记但删除"请优先关注…"文案，改为 Claude Code 风格的无指令标记 | **正确方向** — 上下文标记应该是结构信息，不是 LLM 行为指令 |
| ✅✅ 对齐 Claude Code | 移除整个话题切换检测机制，改用纯 token 阈值的 auto-compact | **最优解** — 但改动量大，需要重构 context_manager |

**推荐执行顺序**：先 ⚠️（应急），再 ✅（改标记），长期考虑 ✅✅（架构对齐）

### 问题 2（原报告）：get_session_context 工具误用

**是否必须修改？** ❌ **不需要修改代码**

**根因审查**：这不是系统 Bug。LLM (qwen3.5-plus) 将用户说的"项目代号"与系统环境信息混淆，
调用 `get_session_context` 获取了系统项目名 "OpenAkita"。这是 **模型理解能力的问题**，
不是工具定义的问题。

- 在同一对话中，T02（回忆名字）PASS、T04-T05（计算）PASS，说明模型大部分时候能正确引用上下文
- T03 失败是因为"项目代号"这个词语在 system prompt 中也存在（OpenAkita 项目），模型产生了歧义
- 换更强的模型（如 Claude 系列）很可能不会出现此问题

**原修复方案评价**：在 system prompt 加"用户陈述优先于工具查询"的规则 — **无害但无用**。
强模型不需要这条规则就能正确理解，弱模型即使看到规则也未必遵守。这种做法会让 system prompt
越来越臃肿，Claude Code 的设计中**没有这类补丁式规则**。

**建议**：归类为**模型局限**，不做代码修改。如果反复出现，考虑升级 LLM 端点。

### 问题 3（原报告）：项目任务列表 API 端点缺失 GET

**是否必须修改？** ❌ **不需要修改，这是测试脚本的 Bug**

**根因审查**：

正确的任务列表端点是 `GET /api/orgs/{org_id}/tasks`（跨项目聚合，支持 `?project_id=` 过滤）。
测试脚本错误地使用了 `GET /api/orgs/{org_id}/projects/{project_id}/tasks`，
该路径只有 `POST`（创建任务）。

实际验证：使用正确端点后，数据完全正常 —

| 组织 | 任务数 | 示例状态 |
|------|--------|----------|
| 创业公司 | 3 | todo, delivered |
| E2E-FullFlow-Test-0408 | 45 | todo, accepted |
| TaskFlow-Verify-Test | 1 | accepted |
| Regression-Fix-Test | 8 | accepted, cancelled |
| FinalFlow-Test | 1 | accepted |
| E2E-Orchestration-Test | 2 | cancelled |
| 创业公司 (2) | 7 | delivered, in_progress, todo |

前端代码使用的也是 `GET /api/orgs/{org_id}/tasks`，所以**不存在端点缺失问题**。
8 个测试 FAIL 全部是测试脚本使用了错误的 URL。

### 问题 4（原报告）：LLM 先否认再查询的行为

**是否必须修改？** ❌ **不需要**

这是 qwen3.5-plus 模型的行为特征，不是系统问题。在 system prompt 中加"先查历史再回复"
这种规则是**垃圾补丁** — Claude Code 的 system prompt 从不包含这类行为矫正规则。
正确的做法是用更好的模型，或接受弱模型的局限。

---

### 问题 5（原报告）：组织编排节点全部 idle

**是否必须修改？** ❌ **正常行为，不需要修改**

command 是异步执行的，5 秒内可能还没有分配到具体节点。节点从 `idle` → `working` 的
状态切换发生在 OrgRuntime 将消息路由到节点并开始 LLM 调用时。在 command 刚提交的 5 秒内，
组织的路由/调度可能还在进行中。

---

### 问题 6（原报告）：无法找到活跃任务进行取消流程测试

**是否必须修改？** ❌ **不需要，测试窗口问题**

任务执行完后自然流转到终态。要测试取消流程应该：创建任务 → dispatch → 立即 cancel。
这是测试方法问题，不是产品 Bug。

## 六、修复计划（仅针对确认的 Bug）

### 应急修复（P0）

**位置**：`src/openakita/core/agent.py:3248`

**改动**：增加通道守卫，Desktop 聊天不触发话题检测（符合 docstring "仅在 IM 通道调用"的设计意图）

### 正式修复（P1）

**位置**：`src/openakita/core/agent.py:3259-3263`

**改动**：移除边界标记中的 `请优先关注边界之后的内容` 指令，改为纯结构标记。
参考 Claude Code 的 `createCompactBoundaryMessage` — 只标记位置，不影响 LLM 推理。

### 长期架构优化（P2，需评审）

**方向**：考虑移除整个 `_detect_topic_change` 机制，对齐 Claude Code 的纯 token
阈值 auto-compact 方案。Claude Code 不做话题检测，对话要么完整保留要么统一压缩，
更简洁可靠。
