### 聊天失败模式分析

4 个失败全部呈现 **同一根因模式**：

```
T01(播种) → T02(回忆名字 ✅) → T03(回忆项目 ❌ 声称"无法知道")
T04(计算 ✅) → T05(追加计算 ❌ 声称"新对话开始")
T06(话题跳转 ✅) → T07(跳回 ❌ 声称"没有记忆") 
T08(更正 ✅) → T09(验证 ❌ 声称"无法知道")
```

**特征**：
- LLM 在**某些轮次**突然声称自己是新对话/无记忆，但在其他轮次又能正确引用历史
- T02 能记住名字但 T03 不能记住项目，说明**上下文在部分轮次被丢失或截断**
- T15 使用 `search_conversation_traces` 工具反而**成功**找到了预算信息
- T20 最终验证 5/5 全对，说明在长对话末尾 LLM 又恢复了完整上下文

**结论**：这是**话题切换边界标记**导致的间歇性上下文遗忘（与上一次测试 v2 结论一致）。

## 四、发现的问题 — 根因分析

### 问题 1 🔴 高严重度：话题切换边界标记导致间歇性上下文遗忘

**影响**: CHAT-03, CHAT-05, CHAT-07, CHAT-09（4/22 = 18% 失败率）

**现象**:
- T02 能记住名字但 T03 不能记住项目（同一批事实）
- T05 说"这是一个新的对话开始"（明显不是）
- T07 说"我没有跨对话的长期记忆"
- T09 说"我无法知道"（但 T08 刚成功更新）
- 但 T15 用 `search_conversation_traces` 工具反而成功
- T19-T20 对话末尾又能正确回忆所有信息

**根因确认（与 v2 报告一致）**:
1. `agent.py:3248` 的话题检测 `_detect_topic_change` 无通道守卫，Desktop 聊天也会触发
2. 话题跳转（如 T06 黑洞→T07 回来）被标记为 NEW 话题，注入 `[上下文边界] ...请优先关注边界之后的内容`
3. 此指令导致 LLM 忽略边界前的消息（T01-T05 的事实）
4. 后续轮次 `_compress_across_boundary` 以 25% 比例压缩边界前消息，进一步丢失
5. **间歇性表现**是因为 LLM 有时能"突破"指令限制（如 T20），但在大多数情况下会遵从"优先关注边界之后"

**对比 Claude Code 设计**: Claude Code 使用纯 token 阈值的 `compactConversation`，完全没有话题检测机制。compact boundary 是纯结构标记，不包含行为指令。

### 问题 2 🟡 中严重度：Pool 模式 LLM debug 日志缺失

**影响**: 无法审计主聊天的 System Prompt 结构

**现象**:
- `data/llm_debug/` 仅包含 org 节点日志
- 主聊天（Pool 模式）在 820 秒测试期间产生 0 个 debug 文件
- Brain._dump_llm_request() 在 Pool 模式下未生效

**根因推测**:
- Pool 模式通过 `AgentInstancePool` 创建独立 Agent 实例
- 这些实例的 Brain 可能使用了不同的配置路径
- 或者 `_dump_llm_request` 中的 `settings.project_root` 在 Pool 实例中指向了不同目录
- 需要检查 `agent_pool.py` 中 Brain 初始化逻辑

### 问题 3 🟢 低严重度：`/api/health/check` 返回 405

**影响**: SYS-02 一项

**现象**: GET `/api/health/check` 返回 405 Method Not Allowed

**根因推测**: 该端点可能只接受 POST，或不存在此路径。需确认 API 注册。

```python
# 修复前
if session and len(session_messages) >= 4:
    topic_result = await self._detect_topic_change(...)

# 修复后
if (session and len(session_messages) >= 4 
    and getattr(session, 'channel', '') == 'im'):
```

**预期效果**: Desktop/API 聊天不再注入话题边界标记，上下文完整保留
